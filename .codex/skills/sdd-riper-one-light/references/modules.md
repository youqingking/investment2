# On-Demand Modules

只在命中场景时加载对应模块，避免常驻耗 token；所有模块都服从 `No Spec, No Code` 与 `No Approval, No Execute`。

## Deep Planning

适用场景：需求模糊、架构设计、跨模块重构、迁移、长链路任务。

执行要点：

- 先收敛目标、边界、约束，写回 spec。
- 先用模型自己的话复述任务理解，确认核心目标一致。
- 如有必要，给 2-3 个方案并说明取舍。
- 计划保持短，但覆盖：`files`、`interfaces`、`order`、`risks`、`validation`。
- 重点不是展开 phase，而是做一次足够清晰的 checkpoint summary。
- 输出 summary/plan 后暂停，等待用户批准，再进入实现。

## Debug

适用场景：报错复现不稳定、根因未知、链路过长、怀疑多处共同导致。

执行要点：

- 先复现，再缩小范围，再形成假设，再验证，不盲改。
- 每个排查 checkpoint 前先重申当前要验证的核心问题，避免漂移；不要在无决策变化时机械重复。
- 优先收集最有区分度的日志、调用链、输入条件。
- 将根因假设、排查结论、拟议修法写入 spec，再请求执行批准。
- 若需要临时探针或额外日志，控制在最小范围，并在收尾时清理或说明。
- 结论至少回答：`根因是什么`、`为什么现在修法有效`、`还有什么未覆盖`。

## Review

适用场景：用户要求评审、任务收尾、风险较高的改动、准备提交前。

默认从 4 个点快速评估：

- `Completion`：目标是否达成。
- `Fidelity`：实现是否偏离 spec / 当前 checkpoint。
- `Quality`：代码正确性、可维护性、测试/验证是否足够。
- `Risk`：残余风险、回归点、后续建议。
- 评审时优先检查：最终实现是否仍然对齐最初复述的核心目标。

输出保持短，并把结论反向同步到 spec；只有高风险时再展开。

## Multi-project

适用场景：多仓、monorepo、前后端联动、跨子项目任务。

执行要点：

- 先声明 `active_project` 与当前改动范围，并写入 spec。
- 先复述这次跨项目任务的核心目标和边界，再决定是否进入 `cross`。
- 默认 `local`，仅在任务明确需要时扩展到 `cross`。
- 跨项目时记录：改了哪些项目、接口如何对齐、依赖顺序是什么。
- 切换项目或进入跨项目实现前，回读该项目相关 spec / codemap 区块。
- 跨项目 checkpoint 获批后再执行。
