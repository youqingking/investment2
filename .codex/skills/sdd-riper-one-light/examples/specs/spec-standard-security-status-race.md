# Spec: 外部协作图片安全状态时序冲突修复

## Goal
- 要解决什么问题：
  外部协作场景发送图片文件时，文件最终实际命中安全拦截，但上传阶段先错误回写 `no_check`，随后又回写 `deny`，导致客户端可能先消费错误状态并在下载链路放行。
- 验收结果：
  同一文件在该场景下不再先出现 `no_check` 再出现 `deny` 的冲突时序；最终只对外呈现正确的安全状态。

## Scope
- In:
  `service/security/ContentSecurityManager.java`
  `service/security/SecurityStatusHandler.java`
  `service/file/FileMetadataManager.java`
- Out:
  客户端代码
  下载链路服务端拦截策略变更
  与本问题无关的规则策略调整

## Facts / Constraints
- 已确认事实：
  同一文件在一次线上样本中先被两次回写为 `no_check`，约 1 秒内又被两次回写为 `deny`。
  审批中间态最终对外表现为拒绝态。
  下载链路不做服务端二次校验是既有产品设定，当前问题核心是上传阶段错误下发了过早的 `no_check`。
  某个旧降级分支会把记录降为 `NO_CHECK` 并立即回写文件状态。
  新的内部安全检测链路仍会继续异步投递，导致后续回调又把状态修正为拒绝态。
  当前同一次状态更新存在双写：一条来自统一更新入口，另一条来自处理器内部的直接回写。
- 技术/业务约束：
  保持现有“下载链路服务端不校验”的特性不变。
  不能粗暴删除旧兼容逻辑，避免影响仍依赖旧链路的历史场景。
  代码修改前需先获得明确批准。
- 已知风险：
  如果直接移除旧降级逻辑，可能波及仍在线上的历史兼容路径。
  如果只修双写、不修状态来源冲突，问题仍会存在。

## Restated Understanding
- 我理解当前任务是：
  修复一次真实线上状态竞争问题，避免文件在外部协作图片场景中先暴露错误安全状态，再被后续回调纠正。
- 当前核心目标是：
  杜绝“先 `no_check`、后 `deny`”这类错误时序，对外只呈现最终正确状态。
- 当前边界是：
  只处理上传阶段的状态冲突和重复回写，不改客户端，不改下载链路策略。
- 暂不处理：
  非当前问题相关的规则扩展、状态定义调整、客户端消费逻辑改造。

## Checkpoint Summary
- 当前任务理解：
  这是旧兼容降级逻辑与新安全检测链路叠加后产生的状态竞争问题。
- 当前核心目标：
  消除早到的错误 `no_check`，并收敛同一次状态更新的重复回写。
- 当前进度：
  已通过日志样本和代码阅读确认现象、来源链路和双写位置。
- 下一步 1:
  将新检测链路与旧降级逻辑解耦，避免提前回写 `no_check`。
- 下一步 2:
  视影响面决定是否增加状态机保护，避免安全检测过程中的逆向降级。
- 下一步 3:
  收敛重复回写逻辑，保留单一可信更新入口。
- 涉及文件 / 模块：
  `ContentSecurityManager`、`SecurityStatusHandler`、`FileMetadataManager`
- 风险：
  需要区分新检测链路与旧兼容链路，不能一起改坏。
- 验证方式：
  复现外部协作图片上传场景，确认日志中不再先出现 `no_check` 后出现 `deny`，且单次状态不再双写。
