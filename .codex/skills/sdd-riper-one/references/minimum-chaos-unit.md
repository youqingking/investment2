# Minimum Chaos Unit

最小混沌单元是可以交给大模型独立推进、同时能被人类低成本验收和回炉的任务单位。

## 判断标准

一个好的任务单元应满足：

- 目标清楚：一句话说清这轮要解决什么。
- 边界可见：In / Out 明确，知道哪些不能顺手改。
- 上下文足够：相关代码、需求、约束、历史决策可被定位。
- 结果能验：有测试、日志、运行结果、截图或人工验收标准。
- 失败可局部重来：失败不会推倒整件事。
- 模型有自主空间：不是逐行遥控，允许模型选择实现路径。

## 拆分流程

1. 先收敛最终目标：这件事最终要达成什么。
2. 列出混沌来源：未知代码、未知需求、跨模块、风险、外部依赖。
3. 用 codemap/context 降低混沌：先看地形，再拆任务。
4. 提出候选任务单元：每个单元都要有目标、边界、验证和回炉方式。
5. 与用户确认优先级和执行顺序。
6. 把用户选择写入 spec，再进入 Plan。

## 常见拆法

- 按代码链路拆：入口 -> 核心逻辑 -> 存储/外部系统 -> 展示/调用方。
- 按风险拆：先兼容/只读/日志，再状态变更，再写路径。
- 按验收拆：先能验证的最小行为，再扩边界。
- 按项目拆：provider 先于 consumer，契约先于实现。

## 不能进入 Plan 的信号

- 最终目标还在变。
- 当前任务单元无法独立验收。
- Scope 外影响不可控。
- 失败后不知道如何回炉。
- 用户无法选择方案或风险接受方式。

这时应继续 Research / Innovate，或补 codemap/context。
